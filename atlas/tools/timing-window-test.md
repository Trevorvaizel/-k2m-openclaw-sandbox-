# Timing Window Test

> Strong ideas still die when the surrounding world is not yet ready to metabolize them.

## Purpose
Use the timing window test to judge whether a proposed innovation is early, late, or precisely aligned with the enabling conditions around it.

## Process
Check:
1. technical readiness
2. user pain intensity
3. trust readiness
4. regulatory tolerance
5. infrastructure support
6. narrative legibility

Then ask which condition is missing and whether waiting, reframing, or narrowing the wedge would improve timing.

## Result
- good ideas stop being misdiagnosed as weak products when timing is the real issue
- teams can separate category education from product iteration
- window mismatch becomes discussable

## When To Use
Use this before scaling a product that feels directionally right but is not being absorbed yet.

## Failure Mode
Using timing as an excuse for untested demand or poor execution.
