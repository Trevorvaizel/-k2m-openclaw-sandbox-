# K2M Touchpoint Territory Audit
## Every Contact Surface As Belief Terrain

*Status: subordinate touchpoint-audit template - reconciled to PRD v0.2 on 2026-04-10*

---

## Purpose

This document defines how K2M should audit touchpoints across penetration, sales, retention, referral, and expansion.

It is not a standalone operating contract.
Use it only under the doctrine and execution hierarchy:

1. `canon/` for philosophy
2. `k2m-belief-operations-execution-prd.md` for implementation contract
3. `k2m-belief-operations-os.md` and runtime-state docs for operating truth

If this template conflicts with those documents, this template loses.

Touchpoint audits should only be treated as valid when the audience already has Phase 0 Product Truth coverage and the audit outputs can be routed into the last-mile cascade: content brief, distribution action, friction fix, retention intervention, referral asset, or map update.

The governing idea is simple:

every touchpoint either builds belief, erodes belief, or wastes attention.

---

## Why This Matters

K2M does not sell a neutral utility.
It sells movement across uncertainty, trust, and identity.

So every touchpoint should be evaluated for:

- what belief it reinforces
- what doubt it triggers
- what friction it adds
- who owns it
- what evidence tier supports the conclusion
- what disconfirming evidence weakens the conclusion
- what external metric should move if the repair is real

---

## Audit Questions

For every touchpoint, ask:

1. What belief should this touchpoint strengthen?
2. What belief does it currently weaken or confuse?
3. What friction is introduced here?
4. What trust layer is active here?
5. What proof is visible here?
6. Who owns this surface?
7. What should happen next if this touchpoint works?
8. What evidence tier supports this diagnosis?
9. What would disconfirm the diagnosis?
10. Which external metric should improve if the repair is correct?

---

## Core Audit Schema

Use this row structure for every touchpoint audit.

| Field | Meaning |
|---|---|
| Touchpoint | the specific interaction surface |
| Territory | penetration, sales, retention, referral, expansion |
| Border | optional border if it sits between territories |
| Audience | who experiences it |
| Current objective | what this touchpoint is supposed to do |
| Belief built | the belief it should strengthen |
| Belief eroded | the belief it currently weakens or undermines |
| Trust layer | suspicion, curiosity, consideration, verification, commitment, advocacy |
| Friction type | cognitive, emotional, social, logistical, financial, technical |
| Friction details | what makes progress harder |
| Proof visible | testimonial, artifact, community, live demonstration, process clarity |
| Owner team | who is responsible |
| Evidence tier | `E0` to `E5` for the diagnostic confidence |
| Disconfirmation note | what evidence could overturn the diagnosis |
| Current metric | observable signal of performance |
| External success metric | usually cohort application rate, DM reply rate, or a downstream conversion/retention metric |
| Severity | low, medium, high |
| Downstream action type | content brief, friction fix, retention intervention, referral asset, map update |
| Next repair action | what should change |

## Touchpoint Inventory By Territory

### Penetration

Typical touchpoints:

- ad
- short video
- social post
- creator collaboration
- landing page
- waitlist page
- school outreach message
- webinar / event invitation
- email capture form

Typical beliefs built:

- this problem is real
- people like me are not broken
- there is a better way to think through this

Typical friction:

- category confusion
- abstract copy
- weak proof
- wrong awareness-stage framing

Typical owner:

- growth / brand / narrative team

### Sales

Typical touchpoints:

- inquiry reply
- FAQ page
- application form
- booking page
- sales call
- offer page
- pricing / scholarship page
- checkout
- confirmation email

Typical beliefs built:

- this is trustworthy
- this is specific to my problem
- this will not waste my time

Typical friction:

- trust gaps
- unclear promise
- unclear mechanism
- process ambiguity
- financial fear

Typical owner:

- sales / admissions / founder / operations

### Retention

Typical touchpoints:

- welcome email
- onboarding form
- Discord invitation
- first live session
- week-one assignment
- office hours
- milestone check-in
- artifact feedback
- graduation moment

Typical beliefs built:

- I belong here
- I can actually do this
- this is changing how I think

Typical friction:

- delayed first proof
- onboarding overload
- unclear next step
- weak community connection
- low psychological safety

Typical owner:

- cohort operations / learning design / community

### Referral

Typical touchpoints:

- graduation share moment
- artifact showcase
- alumni follow-up
- referral ask
- ambassador invitation
- alumni community post
- success-story capture

Typical beliefs built:

- this transformation is real
- this is part of my identity now
- I want others to cross this bridge too

Typical friction:

- awkward ask
- weak share assets
- no clear proof object
- no easy bridge into adjacent clusters

Typical owner:

- community / growth / alumni operations

### Expansion

Typical touchpoints:

- post-program check-in
- advanced offer invitation
- alumni workshop
- case study request
- school introduction request
- parent / teacher recommendation path

Typical beliefs built:

- the relationship continues
- deeper engagement is worthwhile
- my proof can help others enter

Typical friction:

- no next step
- no visible ladder
- proof not captured
- alumni energy decays

Typical owner:

- community / partnerships / growth

## Border Touchpoints

The most dangerous surfaces often sit between territories.

### Penetration -> Sales

Examples:

- CTA from content to landing page
- landing page to application form
- waitlist to booking

Common failure:

- belief created, but trust not transferred

### Sales -> Retention

Examples:

- purchase confirmation
- onboarding instructions
- Discord join flow

Common failure:

- commitment made, but no immediate confidence

### Retention -> Referral

Examples:

- graduation
- artifact completion
- alumni share prompt

Common failure:

- transformation happened, but no advocacy path exists

### Referral -> Penetration

Examples:

- public artifact share
- alumni testimony
- school introduction

Common failure:

- proof exists, but it does not cross into new clusters

## Friction Taxonomy

Use this vocabulary when diagnosing touchpoints.

### Cognitive friction

- unclear message
- too much information
- no visible next step

### Emotional friction

- fear
- shame
- uncertainty
- lack of safety

### Social friction

- fear of looking foolish
- fear of peer or parent judgment
- lack of legitimacy

### Logistical friction

- timing
- scheduling
- long forms
- too many steps

### Financial friction

- unclear price
- unclear value
- unclear aid path

### Technical friction

- broken links
- form issues
- community access issues

## Ownership Rules

Every touchpoint should have:

- one primary owner
- one reviewer if it is high-impact
- one linked Notion record if it materially affects the map

### Rule

If no team owns a touchpoint, that touchpoint is already a liability.

---

## Audit Cadence

Minimum cadence:

- penetration surfaces: monthly
- sales surfaces: monthly
- onboarding and first-value surfaces: every active cohort cycle
- referral and expansion surfaces: every cohort completion cycle
- high-friction border surfaces: weekly while under repair

---

## Required Outputs

Each touchpoint audit should produce:

- touchpoint inventory
- belief built / belief eroded map
- friction ranking
- ownership map
- evidence-tiered diagnostic notes
- disconfirmation notes for major claims
- repair priority list
- border-risk summary
- routed downstream actions tied to the last-mile cascade

---

## Implementation In Notion And ClickUp

### Notion

Store:

- the reviewed audit summary
- cross-touchpoint patterns
- canonical belief implications
- evidence tier and disconfirmation note on major conclusions
- linked Product Truth / Belief Map / Topology Map records where the repair changes canonical understanding

### ClickUp

Store:

- repair tasks
- owner assignments
- workflow changes
- due dates
- external metric linkage where the touchpoint repair is expected to affect cohort application rate, DM reply rate, conversion, retention, or referral

### Suggested task type

- `Border Fix`
- `Experiment`
- `Asset Build`
- `Content Production`
- `Product Friction Fix`
- `Retention Intervention`
- `Referral Asset`

### Runtime note

Current repo evidence supports Notion, ClickUp, and the repo-local MCP server as the real operating surfaces for this work.

- Notion is where reviewed touchpoint patterns should connect back to Product Truth, Belief Maps, and Topology Maps.
- ClickUp is where concrete repairs should run.
- The MCP layer is the routing/orchestration layer for audits and resulting tasks.
- n8n and distribution automation remain target architecture, not a verified live touchpoint-audit runtime in this repo.

## Minimum Viable Audit

If K2M wants the smallest useful version, audit these first:

1. first-touch landing page
2. application / booking path
3. purchase-to-onboarding handoff
4. Discord join flow
5. first live session
6. artifact completion moment
7. graduation / referral ask

That will reveal most of the meaningful cracks.

## Canonical Rule

Every touchpoint is territory.

If a touchpoint changes belief, trust, or momentum, it belongs in this audit system.
