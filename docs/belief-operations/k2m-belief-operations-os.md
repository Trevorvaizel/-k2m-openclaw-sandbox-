## The Governing Principle

K2M cannot build its growth system the way most companies do.

Most companies can separate:

- product
- marketing
- sales
- retention
- referral

K2M cannot do that cleanly because the product itself is already a belief-update engine.

The cohort:

- names an invisible problem
- gives psychological safety
- stages identity movement
- produces proof artifacts
- creates community ties
- turns transformation into public language

So the right model is not:

`marketing -> sales -> retention -> referral`

The right model is:

`product truth -> territory truth -> penetration -> sales -> retention -> referral`

The product is not one continent among the others.
It is the mountain range around which the other territories are built.

And the market is not just a belief field.
It is also a network.

That means K2M has two upstream truths to fix before optimizing messages:

- **product truth:** what the product actually is
- **territory truth:** what the market currently believes, and how those beliefs actually travel

This operating system now assumes the runtime defined in PRD v0.2:

- cognition on Claude Code
- sensing and event loops on n8n
- orchestration via MCP hosted for K2M use on Railway
- canonical maps and reviewed truth in Notion
- execution moves in ClickUp

The belief-ops system is therefore not "documentation about marketing."
It is an operational product whose loop only counts as working when map changes reach published content, product fixes, retention interventions, and referral assets.

## Layer 0: Product Truth

This layer is upstream of all mechanics.
It must be fixed before any workflow is built.
Per the PRD, Phase 0 requires Product Truth coverage for all planned audiences before deep implementation begins on the student slice.

| Layer | Current canonical statement | Role |
|---|---|---|
| **Philosophy** | AI changes the conditions of intelligence and expertise | Internal worldview |
| **Product kernel** | K2M builds identity-safe AI agency under uncertainty | Design anchor |
| **Product identity** | I am someone who thinks with AI | Achieved self |
| **Bridge identity** | I can think with AI, decide with clarity, and own the result | Recognizable self before purchase |
| **Category belief** | Nobody taught you how to think through uncertainty with AI | Market awakening |
| **Product belief** | K2M teaches that way of thinking through habits, structure, psychological safety, and proof | Product claim |
| **Student beachhead payoff** | I decided my next chapter. It was not decided for me | Student conversion trigger |

### Layer 0 outputs

Before any territory work begins, K2M must have:

- one product identity
- one bridge identity per audience
- one category belief per audience
- one product belief per audience
- one defining payoff line per beachhead
- one proof object or artifact per core offer
- a review state and evidence tier for each Product Truth record
- explicit notation of which non-student audiences are still hypotheses

### Layer 0 failure mode

If this layer is unclear, every downstream system breaks:

- penetration creates interest without anchor
- sales verifies the wrong promise
- retention deepens the wrong identity
- referral spreads language that does not map to the real product

## Layer 1: Territory Truth

This is the layer the manifestos insist most operators skip.

They optimize messages before mapping the network those messages have to travel through.

For K2M, territory truth has two inseparable parts:

### 1. Belief architecture

What the audience currently believes about:

- their problem
- available solutions
- their own capability
- what change costs
- whether people like them successfully do this

The typed structure for this layer is the PRD's Four Forces schema:

- **Push:** pain with the current state
- **Pull:** attraction toward the possible new state
- **Anxiety:** fear of the new state
- **Habit:** comfort with the status quo

### 2. Network topology

How those beliefs move through the population:

- which clusters exist
- which nodes bridge those clusters
- which mavens hold credibility inside each cluster
- which connectors carry ideas across weak ties
- which threshold bands are ready to move
- where a cascade is likely to stall

### The correction

The current OS mentioned network mapping inside Penetration.
That is not strong enough.

The better rule is:

> **Do not optimize messages before mapping the network.**

Message quality matters.
But in the canonical documents, belief cascades do not happen because a message is merely well written.
They happen because the message reaches:

- the right threshold-zero nodes
- through the right bridges
- at the right timing window
- with enough trust weight to move the next threshold band

### Layer 1 outputs

Before any major message optimization, K2M should have:

- belief audit
- Four Forces map
- awareness-stage map
- self-verdict map
- failed-promises map
- cluster map
- bridge-node map
- maven map
- connector map
- threshold-band hypothesis
- cascade risk hypothesis
- evidence tier tagging on every major conclusion
- explicit disconfirmation notes where the evidence is thin or contradictory

### Layer 1 failure mode

If this layer is weak:

- content circulates inside existing supporters but does not penetrate
- shares look healthy but do not cross weak ties
- campaigns reach the wrong threshold band
- the system misdiagnoses distribution failure as copy failure

### Layer 1 governing question

Before asking "how do we improve this message?" ask:

> **Who needs to carry this message across which bridge for the next threshold to move?**

## The Four Territories

The cartographer manifestos describe four linked territories:

- penetration
- sales
- retention
- referral

In the operating product, these four territories sit inside a larger 10-skill system.
The extra skills handle upstream truth, intelligence maintenance, authority translation, and orchestration:

- Product Truth
- Belief Audit
- Network Topology Audit
- Authority Intelligence
- Live Market Intelligence
- System Orchestrator

For K2M, each territory must be defined by:

- the belief job
- the customer job
- the diagnostic tools
- the required outputs
- the success metrics
- the handoff to the next territory

## Territory 1: Penetration OS

### Core job

Enter an existing belief system and make the real problem impossible to unsee.

K2M penetration is not mainly reach generation.
It is category formation.

### Strategic objective

Move the market from:

- "I am confused and that is just how things are"

to:

- "I am not broken; nobody taught me how to think through uncertainty with AI"

### Canonical tools

- belief audit
- identity map
- awareness-stage map
- network topology map
- timing window map
- sentiment probes

### Required outputs

- audience belief brief
- failed-promises map
- self-verdict map
- cluster map
- maven map
- connector map
- bridge-node map
- threshold-band map
- category belief statement
- wedge insight
- product belief statement
- beachhead segment definition
- threshold-zero node hypothesis
- cascade path hypothesis
- content architecture by awareness stage
- first content briefs routed from map insights into real channels
- channel and format selection hypotheses tied to topology, not taste

### K2M penetration artifacts

- category belief posts
- wedge videos
- territory-first landing pages
- audience-specific social proof
- beachhead audience narrative kits
- language bank from comment threads and interviews
- bridge-node outreach list
- threshold-zero creator or community seed list

### Penetration metrics

- problem-recognition rate
- saves-to-shares ratio
- bridge-node share ratio
- cluster-crossing ratio
- maven-engagement ratio
- threshold-zero activation rate
- qualified opt-in rate
- audience-language match
- comment vocabulary shift from confusion to recognition
- cohort application rate

### K2M-specific penetration rule

K2M must build category belief before product belief.
And it must map network topology before trying to optimize content volume.

Do not lead with:

- cohort mechanics
- tool lessons
- prompt engineering
- the full transformation claim

Lead with:

- the invisible gap
- the asymmetry
- the self-verdict being misdiagnosed

And before scaling the message, answer:

- which cluster is the beachhead
- which nodes can bridge to the adjacent cluster
- which threshold band is being targeted first
- what proof travels well across that bridge

### Penetration-to-sales handoff

A prospect is ready to cross into sales when they can already say:

- the problem is real
- the problem is about thinking, not just access
- people like me may be able to solve it

## Territory 2: Sales OS

### Core job

Convert belief into commitment through trust verification.

Sales is not persuasion theater.
It is the process by which the prospect tests whether the map they formed in penetration matches the actual territory of the product.

### Strategic objective

Move the prospect from:

- "This sounds true in general"

to:

- "This specific K2M offer can safely help someone like me"

### Canonical tools

- trust gradient audit
- friction walk
- objection map
- proof sequence map
- expectation alignment audit

### Required outputs

- sales-page trust map
- FAQ by trust layer
- objection matrix
- friction log
- proof stack
- expectation contract
- 24-72 hour post-purchase handoff plan
- product friction cascade linking diagnosed issues to GitHub or ClickUp fixes

### K2M sales artifacts

- landing pages aligned to awareness stage
- FAQ pages that answer identity fear, not just logistics
- proof objects from prior cohorts
- "what K2M is not" integrity statements
- call scripts or DM scripts
- purchase-to-onboarding bridge assets

### Sales metrics

- page-to-application conversion
- application-to-payment conversion
- DM reply rate
- sales friction drop-off points
- expectation mismatch rate
- refund-request themes
- time from commitment to first onboarding action

### K2M-specific sales rule

The sale should verify four things:

- the category belief
- the product belief
- the safety of the environment
- the credibility of the proof path

If the buyer is still asking whether the category is real, they are not in sales yet.
They are still in penetration.

### Sales-to-retention handoff

The moment of purchase creates a dangerous gap.

K2M must explicitly manage the first 24-72 hours after payment:

- what they expected
- what happens next
- what first win looks like
- what emotional friction will likely appear

If this handoff is weak, the trust built in sales collapses inside onboarding.

## Territory 3: Retention OS

### Core job

Deepen the customer's identity and move them from shallow participation to durable adoption.

Retention is not "prevent churn."
Retention is identity deepening plus value realization.

### Strategic objective

Move the customer from:

- "I joined a cohort"

to:

- "I am becoming the kind of person who thinks with AI"

and eventually:

- "This way of thinking is now part of how I see myself"

### Canonical tools

- friction walk across the full cohort journey
- defining-moment design
- value realization map
- community tie-density audit
- identity-language audit
- dropout and stagnation analysis

### Required outputs

- first-value threshold definition
- week-by-week identity deepening map
- Discord journey friction map
- defining moments calendar
- recovery interventions for stalled participants
- artifact completion map
- community tie strategy
- defining moments tied to real student proof, not hypothetical journey maps

### K2M retention artifacts

- onboarding welcome sequences
- week-by-week prompt architecture
- live-session scripts
- milestone rituals
- artifact templates
- graduation rituals
- public showcase moments

### Retention metrics

- onboarding completion
- first meaningful action time
- weekly participation depth
- artifact completion
- identity-language shift in reflections
- peer interaction density
- zone progression
- post-program confidence and authorship signals

### K2M-specific retention rule

The product must produce visible proof fast enough to stop the belief gap from reopening.

Retention is strongest when students can feel, not just be told, that:

- AI is now accessible
- their thinking matters
- their context changes outcomes
- their choices are defensible

### Retention-to-referral handoff

Referral should not be requested from average satisfaction.

It should emerge when three conditions exist:

- the learner has real proof
- the learner can narrate the transformation
- sharing K2M would make them look generous, intelligent, or ahead

## Territory 4: Referral OS

### Core job

Turn transformation into advocacy that crosses weak ties into new clusters.

Referral is not a coupon mechanic.
It is an act of identity expression.

### Strategic objective

Move the customer from:

- "This helped me"

to:

- "People like me should know this"

and then:

- "Sharing this says something true about who I am"

### Canonical tools

- peak-experience map
- referral story audit
- bridge-node map
- weak-tie map
- weak-tie asset design
- advocacy friction walk

### Required outputs

- referral trigger inventory
- advocacy moment design
- shareable proof assets
- bridge-crossing story kits
- alumni amplification map
- weak-tie distribution hypotheses
- cluster-to-cluster referral paths
- maven-ready proof assets
- referral assets that feed the next penetration pass with student proof

### K2M referral artifacts

- graduation artifacts worth sharing
- public transformation stories
- short social scripts students can actually use
- alumni case studies
- cohort showcase moments
- parent-facing and peer-facing proof assets

### Referral metrics

- organic referral rate
- referral-to-enrollment conversion
- share rate of artifacts
- alumni amplification rate
- weak-tie spread indicators
- bridge-crossing conversion rate
- quality of referred leads

### K2M-specific referral rule

Do not reduce referrals to incentives first.

First design for:

- identity pride
- generosity
- social proof
- public language
- bridge-node portability

Only after the organic mechanism is working should any incentive be considered, and even then it must not cheapen the social meaning of the referral.

## The Border Diagnostics

The most dangerous failures happen at the borders between territories.

### Border 1: Penetration -> Sales

**Failure pattern:** category belief is built, but the offer asks for a stronger commitment than the trust gradient supports.

**Symptoms:**

- strong engagement, weak conversion
- people agree with the idea but hesitate on the offer
- comments sound resonant but applications stay low

**Diagnostic question:**

Have we built category belief and credibility, but not enough competence and integrity trust?

### Border 2: Sales -> Retention

**Failure pattern:** the promise made in conversion is not matched by the first lived experience of the product.

**Symptoms:**

- drop-off after payment
- silent disengagement in the first week
- regret framed as "timing" or "overwhelm"

**Diagnostic question:**

What expectation did sales create that onboarding failed to honor immediately?

### Border 3: Retention -> Referral

**Failure pattern:** customers are satisfied enough to stay, but not transformed enough to advocate.

**Symptoms:**

- okay completion
- low public sharing
- high praise in private, low referral in public

**Diagnostic question:**

Did we produce satisfaction, or did we produce identity pride and proof?

### Border 4: Referral -> Penetration

**Failure pattern:** customers share, but the share does not cross clusters or build belief in new audiences.

**Symptoms:**

- referrals stay inside the same social circle
- shared artifacts get likes but no qualified interest
- alumni sharing does not produce new thresholds

**Diagnostic question:**

Are our advocacy assets good enough for existing supporters, but too weak to cross weak ties into adjacent clusters?

### Border 0: Product Truth -> Territory Truth

**Failure pattern:** the product is clear internally, but the team assumes the market's network will naturally carry the story.

**Symptoms:**

- confidence in the narrative, weak actual spread
- repeated optimization of copy with little distribution change
- good resonance inside the house list, weak movement outside it

**Diagnostic question:**

Have we clarified the product but skipped the work of mapping how belief actually moves through the market?

## The K2M Operating Loop

This is the minimum operating loop for the system.

1. Complete Phase 0 Product Truth coverage for students, teachers, principals, and B2B.
2. Lock the active student Product Truth record with owner, reviewer, relations, and evidence tier.
3. Run belief audit, network topology audit, authority intelligence, and live intelligence for the active audience.
4. Express belief conclusions in Four Forces form and tag every major claim with an evidence tier.
5. Run a disconfirmation pass before any major synthesis is promoted toward canonical state.
6. Define category belief, product belief, wedge, cascade path, and channel/format plan.
7. Produce and publish the first map-grounded content briefs through the current distribution phase.
8. Audit trust gradient and run the sales friction walk across the live product surfaces.
9. Route product and operational fixes into ClickUp and code/system backlogs.
10. Design first-value thresholds, defining moments, retention interventions, and proof capture.
11. Turn proof into referral assets that can cross weak ties and feed the next penetration cycle.
12. Observe border failures, update the map, and refresh coverage/freshness state.

This loop then repeats:

- for the next audience
- for the next cohort
- for the next market condition

The system is not static.
It is a map update discipline.

## Skill Surface And Workflow Contracts

These are the current canonical skills from PRD v0.2.
This document defines their structural role. The PRD remains the source of truth for acceptance criteria and release gating.

### 1. `k2m-product-truth`

Purpose: keep Layer 0 stable, explicit, and review-gated for one audience.

### 2. `k2m-belief-audit`

Purpose: map the current territory in Four Forces form with evidence-linked belief statements.

### 3. `k2m-network-topology-audit`

Purpose: map clusters, bridges, mavens, connectors, threshold bands, and cascade risks before message optimization.

### 4. `k2m-penetration-architecture`

Purpose: convert product truth plus territory truth into wedge design, routing plans, content briefs, and beachhead entry strategy.

### 5. `k2m-sales-friction-walk`

Purpose: diagnose trust gradient and commitment friction across the real student-facing product surfaces.

### 6. `k2m-retention-architecture`

Purpose: define first value, identity deepening, dropout intervention, and proof-capture design.

### 7. `k2m-referral-architecture`

Purpose: turn transformation into weak-tie-crossing advocacy that re-enters penetration.

### 8. `k2m-live-market-intelligence`

Purpose: keep the map current through sensing, evidence normalization, memo production, and timing-window detection.

### 9. `k2m-system-orchestrator`

Purpose: coordinate workflows, border diagnostics, prioritization, and routing between Notion, ClickUp, and downstream systems.

### 10. `k2m-authority-intelligence`

Purpose: translate global authority and borrowed credibility into audience-specific assets without replacing direct local evidence.

### Implementation state

- The 10-skill inventory is canonical.
- The repo-local MCP server already implements orchestration helpers, doctrine scoring, Notion/ClickUp integration, and intelligence collection support.
- Most named skills are **not yet implemented as full independent runtime skills**. The docs must distinguish target architecture from current live capability.

## Build Order

Do not build this system all at once.

Build it in the PRD's decided order:

1. Phase 0 Product Truth for all audiences
2. student Product Truth deep pass
3. student intelligence territories in parallel: belief audit, topology audit, authority intelligence, live market intelligence
4. penetration with first map-grounded content briefs and published content
5. sales with trust gradient audit, friction walk, and routed product fixes
6. retention with first-value and defining-moments architecture
7. referral with proof-led advocacy that feeds the next penetration cycle
8. border diagnostics and cross-territory repair
9. only then promote the next audience

That order matters because K2M is still validating which parts of the student architecture travel unchanged into teachers, principals, and B2B contexts, and because the system only counts as operational when intelligence reaches the last mile.
